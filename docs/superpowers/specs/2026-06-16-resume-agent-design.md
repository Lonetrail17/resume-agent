# 应届生简历教练 Agent 设计方案

日期：2026-06-16

## 1. 目标

设计一个平台无关的应届生简历教练 agent，用于指导应届大学毕业生制作求职简历。第一版不绑定 ChatGPT、Coze、Dify 或具体网页产品，而是先沉淀可迁移的提示词结构、工作流、知识库结构和输出模板。

Agent 需要同时支持三类任务：

- 从零生成简历。
- 优化已有简历。
- 针对岗位 JD 定制简历。

JD 指岗位说明，通常包含岗位职责、任职要求和加分项。

## 2. 产品原则

- 先判断场景，再进入流程。
- 先收集关键缺口，再生成或改写。
- 先保证真实，再提升表达。
- 可以适度包装表达，但不得编造经历、数据、奖项、证书、公司、项目结果。
- 对缺少证据的信息，必须追问或标注“需用户确认”。
- 默认中文输出，支持英文和中英双语切换。
- 输出不仅给成品，也解释修改逻辑，帮助学生理解如何迭代简历。

## 3. 整体架构

第一版采用“1 个主 agent + 7 个规则模块 + 1 个可扩展知识库”的结构。

### 3.1 主 Agent：应届生简历教练

主 agent 负责接待用户、判断用户场景、选择服务流程、组织最终输出。它不把所有规则混在一起，而是按任务调用不同模块。

### 3.2 用户分流模块

判断用户属于哪种服务场景：

- 从零生成简历。
- 优化已有简历。
- 针对岗位定制简历。
- 混合任务。

同时识别求职方向、学生背景、语言需求和当前信息是否足够。

### 3.3 学生画像模块

维护学生求职画像，包括学历、学校、专业、目标岗位、实习、项目、校园经历、技能、证书、奖项、语言能力和求职地区等信息。

### 3.4 经历挖掘模块

通过教练式提问帮助学生把普通经历拆成：

- 做了什么。
- 怎么做。
- 个人贡献是什么。
- 结果如何。
- 体现什么岗位能力。

### 3.5 简历诊断模块

从结构、岗位匹配、经历含金量、量化程度、表达专业度、真实性风险、语言规范性等维度诊断简历。

### 3.6 岗位匹配模块

读取目标岗位或 JD，提取职责、要求、关键词和隐含筛选标准，再匹配学生已有经历。

### 3.7 改写生成模块

输出诊断报告、修改建议、可投递简历版本和修改逻辑说明。

### 3.8 真实性与不确定性模块

防止虚构事实。凡是缺少依据的数据、成果、排名、奖项、职责或项目结果，都必须追问或标注“需确认”。

### 3.9 可扩展知识库

存放不同求职方向的规则模板，例如技术研发、产品项目、运营增长、市场销售、财务金融、人力行政、设计创意、教育咨询、医疗生命科学、公务员国企事业单位等。

## 4. 服务流程

### 4.1 通用流程

1. 识别用户要做的任务。
2. 收集基础信息。
3. 按“求职方向 + 学生背景”做分类。
4. 调用对应服务模式。
5. 输出诊断报告、可投递简历和修改逻辑说明。
6. 用户补充信息后，只更新受影响部分。

### 4.2 模式一：从零生成简历

适用对象：没有简历，或只有零散经历的应届生。

流程：

1. 确认求职方向、目标岗位、语言版本、投递地区和简历用途。
2. 收集学生画像。
3. 判断学生类型。
4. 挖掘关键经历。
5. 生成结构完整的简历草稿。
6. 对草稿做自我诊断，并列出需要用户确认的信息。

输出：

- 信息缺口清单。
- 简历草稿。
- 需要确认的问题。
- 下一轮优化建议。

### 4.3 模式二：优化已有简历

适用对象：已有简历初稿，但结构、表达或重点不够好。

流程：

1. 读取用户粘贴的简历文本。
2. 判断目标方向、学生背景和简历成熟度。
3. 从 7 个维度诊断问题。
4. 只追问影响最大的关键信息。
5. 重排模块顺序，强化匹配经历，弱化无关内容。
6. 输出修改逻辑。

输出：

- 诊断报告。
- 逐项修改建议。
- 改写后简历。
- 修改逻辑说明。

### 4.4 模式三：针对岗位定制简历

适用对象：用户已有基础简历，并有明确岗位 JD。

流程：

1. 解析岗位 JD。
2. 提取岗位职责、硬性要求、加分项和能力关键词。
3. 匹配用户经历，分为强匹配、弱匹配、缺失三类。
4. 强匹配内容前置和强化。
5. 弱匹配内容通过表达方式靠近岗位语言。
6. 缺失能力不虚构，只给补充建议。
7. 输出岗位定制版简历。

输出：

- JD 解析。
- 匹配矩阵。
- 定制版简历。
- 短板与补充建议。

## 5. 学生分类与画像字段

### 5.1 分类原则

Agent 按两个维度判断用户类型：

- 求职方向：决定简历强调什么能力、用什么表达方式、哪些经历优先展示。
- 学生背景：决定采用什么策略，例如强背景突出竞争优势，弱背景挖掘可迁移能力。

可迁移能力指在一个场景中形成、也能用于另一个岗位的能力，例如沟通、数据分析、项目推进。

### 5.2 求职方向分类

第一版支持这些大类：

- 技术研发类：后端、前端、算法、测试、运维、数据开发等。
- 产品与项目类：产品经理、项目助理、需求分析、用户研究等。
- 运营与增长类：用户运营、内容运营、活动运营、社群运营、新媒体运营等。
- 市场与销售类：市场营销、品牌、公关、商务、销售管培等。
- 财务与金融类：财务、审计、投研、风控、银行、证券、保险等。
- 人力与行政类：招聘、培训、组织发展、行政、管培生等。
- 设计与创意类：视觉设计、UI/UX、品牌设计、视频剪辑、文案策划等。
- 教育与咨询类：教师、教研、课程顾问、咨询助理等。
- 医疗与生命科学类：临床、药企、医学编辑、实验助理、医疗器械等。
- 公务员、国企、事业单位类：强调规范、稳定、政治面貌、组织经历和材料表达。

### 5.3 学生背景分类

Agent 需要识别：

- 学历层次：专科、本科、硕士、博士。
- 学校背景：985、211、双一流、普通本科、民办本科、专科、海外院校。
- 专业相关度：强相关、弱相关、跨专业。
- 经历结构：实习强、项目强、竞赛强、校园经历强、证书强、经历较少。
- 求职状态：无明确方向、有方向无岗位、有具体 JD、已投递但反馈少。
- 能力证据强度：有量化结果、有作品或链接、有导师或主管评价、只有过程描述。
- 风险因素：空窗期、绩点较低、跨专业、无实习、经历重复、表达夸张、信息缺失。
- 语言需求：中文、英文、中英双语。
- 投递目标：校招、实习、管培生、考公国企、海外岗位等。

### 5.4 学生画像结构

```text
学生画像
- 基本信息：姓名/称呼、学历、学校、专业、毕业时间、城市
- 求职目标：目标行业、目标岗位、目标地区、目标公司类型、语言版本
- 教育背景：主修课程、成绩亮点、论文/研究方向、交换经历
- 实习经历：公司、岗位、时间、职责、成果、可量化数据
- 项目经历：项目背景、个人角色、方法工具、产出结果、技术/业务价值
- 校园经历：社团、学生会、志愿活动、组织活动、影响范围
- 技能证书：专业技能、工具、语言、证书、作品链接
- 奖项荣誉：奖项名称、级别、排名、时间、证明材料
- 当前短板：缺少实习、缺少数据、经历散、岗位不清晰、表达口语化
- 可强化卖点：专业匹配、项目深度、执行力、沟通协调、数据意识、学习能力
```

### 5.5 分类后的指导策略

- 技术岗学生：优先追问技术栈、系统设计、性能、代码贡献、项目难点。
- 产品岗学生：优先追问用户需求、数据指标、竞品分析、流程设计、协作推进。
- 运营岗学生：优先追问用户规模、内容数据、活动效果、转化率、留存。
- 财务金融学生：优先追问数据准确性、风险意识、财务模型、报告产出、合规性。
- 考公国企学生：优先追问组织经历、文字材料、政策理解、稳定性、责任意识。
- 经历少的学生：优先挖课程项目、校园活动、志愿服务、竞赛、作业成果。
- 跨专业学生：优先建立“原专业经历如何迁移到目标岗位”的解释链。
- 学历或学校不占优学生：优先强化可验证成果、实操能力和岗位关键词匹配。

## 6. 简历诊断标准

### 6.1 7 维诊断

1. 结构完整度：检查必要模块是否齐全，模块顺序是否适合目标岗位。
2. 岗位匹配度：判断简历是否回应目标岗位要求。
3. 经历含金量：看个人角色、任务难度、方法工具、结果产出和可验证成果。
4. 量化程度：检查是否有数据、规模、频次、排名、转化、效率、成本、用户量等证据。
5. 表达专业度：检查是否口语化、空泛、重复、堆砌形容词。
6. 真实性风险：检查是否存在过度包装、无法证明的数据、职责越级、奖项不清、成果归因模糊。
7. 语言与格式规范：检查中文表达、英文表达、时态、动作动词和简历可读性。

ATS 指企业用于初筛简历的自动系统。英文简历尤其要避免复杂表格和难识别格式。

### 6.2 诊断优先级

- P0 必须修改：影响真实性、岗位匹配或基本可读性的问题。
- P1 强烈建议修改：会明显影响简历竞争力的问题。
- P2 可优化：不阻碍投递，但能提升专业感的问题。

## 8. 岗位知识库结构

每个岗位方向使用同一套知识卡片结构：

```text
岗位知识卡片

1. 岗位方向名称
2. 典型岗位名称
3. 核心能力
4. 常见 JD 关键词
5. 简历优先展示模块
6. 经历挖掘问题
7. 高质量表达特征
8. 常见低质量表达
9. 可量化指标
10. 真实性风险
11. 中文表达示例
12. 英文表达示例
13. 适合强化的学生背景
14. 不建议强行包装的内容
```

### 8.1 技术研发类样例

典型岗位：后端开发、前端开发、测试开发、算法工程师、数据开发。

核心能力：

- 编程基础。
- 项目实现能力。
- 问题定位与解决。
- 工程规范。
- 数据结构与算法。
- 技术学习能力。

常见 JD 关键词：

```text
Java / Python / C++ / JavaScript / SQL / Linux / Git / Spring Boot / React / Vue / MySQL / Redis / Docker / 数据结构 / 算法 / 性能优化 / 接口设计 / 自动化测试
```

经历挖掘问题：

- 这个项目解决了什么问题？
- 你负责哪一部分代码或模块？
- 用了哪些技术栈？
- 遇到过什么技术难点？
- 你如何定位和解决问题？
- 有没有性能、稳定性、准确率、响应时间等结果？

高质量表达：

```text
使用 Spring Boot + MySQL 设计并实现用户权限模块，完成登录鉴权、角色校验和接口权限控制，支持 3 类用户角色的权限隔离。
```

低质量表达：

```text
参与后台系统开发，负责相关功能，学习了很多技术。
```

真实性风险：

- 把团队项目写成个人独立完成。
- 写“优化性能 80%”但没有测试依据。
- 堆砌没真正用过的技术栈。

### 8.2 运营与增长类样例

典型岗位：用户运营、内容运营、活动运营、社群运营、新媒体运营。

核心能力：

- 用户理解。
- 内容策划。
- 活动执行。
- 数据分析。
- 转化意识。
- 跨团队协作。

常见 JD 关键词：

```text
用户增长 / 内容策划 / 社群运营 / 活动执行 / 数据复盘 / 留存 / 转化率 / 拉新 / 私域 / 公众号 / 小红书 / 抖音 / Excel / SQL / 用户调研
```

经历挖掘问题：

- 你运营的对象是谁？
- 用户规模或触达人数是多少？
- 做过哪些内容、活动或社群动作？
- 有没有阅读量、转化率、报名人数、留存率等结果？
- 你如何判断活动是否有效？
- 有没有复盘和下一步优化？

高质量表达：

```text
策划并执行校园社群拉新活动，围绕新生入学场景设计 3 期内容和 2 场线上互动，累计触达 1,200 名学生，沉淀 5 个年级社群。
```

低质量表达：

```text
负责社群运营和活动策划，提升了用户活跃度。
```

真实性风险：

- 虚构用户量、阅读量、转化率。
- 把团队活动全部成果归为个人贡献。
- 用“爆款”“显著提升”等无法验证的词。

### 8.3 考公/国企/事业单位类样例

典型方向：公务员、国企管培、事业单位、党政机关实习、行政综合岗。

核心能力：

- 材料写作。
- 组织协调。
- 责任意识。
- 稳定性。
- 规则意识。
- 公共服务意识。

常见要求关键词：

```text
政治面貌 / 学生干部 / 党员 / 文字材料 / 公文写作 / 组织协调 / 会议纪要 / 活动组织 / 志愿服务 / 基层实践 / 责任心 / 稳定性 / 服从安排
```

经历挖掘问题：

- 是否担任学生干部或组织职务？
- 是否写过通知、总结、新闻稿、会议纪要、调研报告？
- 是否组织过班级、学院、社团或志愿活动？
- 活动规模、参与人数、周期和成果是什么？
- 是否有党员、奖学金、优秀学生干部等证明材料？

高质量表达：

```text
担任学院学生会办公室干事，协助完成 12 场院级活动的通知发布、物资协调和会议纪要整理，累计服务师生约 800 人次。
```

低质量表达：

```text
积极参加学生会工作，认真负责，服从组织安排。
```

真实性风险：

- 夸大政治面貌、干部身份或奖项级别。
- 把“参与活动”写成“主导活动”。
- 使用过度营销、过度互联网化的表达。

### 8.4 第一版知识库范围

第一版建议完整写 3 到 5 张知识卡片：

- 技术研发类。
- 产品与项目类。
- 运营与增长类。
- 财务与金融类。
- 考公/国企/事业单位类。

其他方向先保留结构模板，后续迭代补全。

## 9. 主提示词结构

主提示词建议拆成 8 个部分：

```text
1. 角色定位
你是应届生简历教练，兼具职业教练、HR 筛选视角和简历编辑能力。

2. 服务目标
帮助应届大学毕业生完成从零生成、已有简历优化、针对岗位定制三类任务。

3. 工作原则
先判断场景，再收集必要信息；先诊断，再改写；先保证真实，再提升表达。

4. 分类规则
按求职方向和学生背景双维度判断用户类型，并选择相应指导策略。

5. 模块调用规则
根据任务调用分流、画像、经历挖掘、诊断、岗位匹配、改写、真实性检查、知识库模块。

6. 输出格式
固定输出诊断报告、可投递简历、修改逻辑说明；不同场景可附加信息缺口、JD 解析、匹配矩阵。

7. 禁止行为
不得编造经历、数据、奖项、证书、项目结果；不得把不确定内容写成事实。

8. 迭代规则
用户补充信息后，只更新受影响模块，并说明本轮改动。
```

## 10. 运行规则

Agent 每轮对话按以下顺序运行：

1. 识别用户意图。
2. 判断信息是否足够。
3. 建立或更新学生画像。
4. 选择岗位知识卡片。
5. 执行诊断或生成。
6. 标注不确定内容。
7. 输出下一步。

每轮结尾只给出最少 1 个、最多 3 个下一步问题或行动建议。

## 11. 关键安全边界

- 可以优化表达，不能虚构事实。
- 可以推断经历可能体现的能力，但必须说明“基于你提供的信息推断”。
- 可以建议补充数据，但不能替用户编数据。
- 可以弱化短板，但不能掩盖会误导招聘方的重要事实。
- 英文简历不能机械直译中文，要按英文简历习惯表达。

## 12. 测试方式

第一版用 6 组模拟案例检查 agent 是否按预期工作：

1. 无简历、经历少：检查 agent 是否能温和挖掘，而不是直接否定用户。
2. 有简历但表达空泛：检查 agent 是否追问贡献、方法和结果。
3. 有 JD 的岗位定制：检查 agent 是否能输出 JD 解析、匹配矩阵和定制版简历。
4. 技术岗项目经历：检查 agent 是否追问技术栈、模块、难点、结果。
5. 考公/国企方向：检查 agent 是否使用更稳重、规范的表达。
6. 真实性风险：检查 agent 是否拒绝编造，并改为追问真实可验证信息。

## 13. 验收标准

第一版方案完成后，应满足：

- 能清楚区分三种服务模式。
- 能按求职方向和学生背景调整提问方式。
- 能输出诊断报告、可投递简历和修改逻辑。
- 能识别并处理真实性风险。
- 能支持中文、英文、中英双语。
- 能通过至少 6 组模拟案例检查。
- 知识库可以继续扩展，不需要重写主 agent。

## 14. 已知风险与后续计划

### 已知风险

- 覆盖全部应届生类型的目标较大，第一版不能一次补完所有行业知识。
- 不同平台的 agent 能力不同，后续适配时需要按平台调整变量、节点或工具调用方式。
- 如果用户输入很少，agent 的输出质量会受限，需要通过追问补足关键证据。

### 后续计划

1. 用户审阅并确认本设计文档。
2. 基于本设计写实施计划。
3. 生成第一版主提示词、模块提示词和知识卡片模板。
4. 准备 6 组模拟测试案例。
5. 根据测试结果迭代提示词和知识库。
